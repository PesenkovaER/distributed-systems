# Лабораторная работа 3.1. Организация асинхронного взаимодействия микросервисов с помощью брокера сообщений

---
## Песенкова Екатерина, ЦИБ-241
## Вариант 15
## Задачи:
1. Подписка на рассылку.  
   Producer отправляет email.  
   gRPC сервис добавляет его в "базу данных" (список в памяти) и возвращает сообщение:  
   "Email добавлен в рассылку".

2. Генерация случайного пароля.  
   Producer отправляет желаемую длину.  
   gRPC сервис генерирует случайный пароль указанной длины и возвращает его.

3. Подсчет гласных.  
   Producer отправляет слово.  
   gRPC сервис подсчитывает количество гласных букв и возвращает число.

---

## Архитектура системы

``` mermaid
flowchart LR

U[Пользователь] -->|Ввод данных| P[Producer]

subgraph RabbitMQ [Асинхронное взаимодействие]
    P -->|Сообщение| Q[(Очередь)]
    Q -->|Получение| C[Consumer]
end

subgraph gRPC [Синхронное взаимодействие]
    C -->|Запрос| S[gRPC Server]
    S -->|Ответ| C
end

C -->|Вывод результата| U
```
## Общая логика работы

1. Пользователь вводит сообщение в Producer  
2. Producer отправляет сообщение в очередь RabbitMQ  
3. Consumer получает сообщение из очереди  
4. Consumer определяет тип задачи по префиксу  
5. Consumer вызывает соответствующий метод gRPC  
6. gRPC сервер обрабатывает запрос  
7. Результат возвращается и выводится пользователю

# ЧАСТЬ 1. Реализация gRPC (синхронное взаимодействие)

- GeneratePassword — генерация пароля  
- CountVowels — подсчет гласных  

Сгенерированы Python-файлы [message_service_pb2.py](message_service_pb2.py) и [message_service_pb2_grpc.py](message_service_pb2_grpc.py) с использованием grpc_tools.

Реализованы методы:

- добавление email в список

- подсчет гласных букв  

Реализован [gRPC клиент](grpc_client.py), который используется для тестирования [сервера](grpc_server.py):

- пользователь выбирает операцию

## Вывод
В ходе выполнения лабораторной работы была реализована система микросервисного взаимодействия, включающая синхронный и асинхронный подходы.
С использованием gRPC реализовано взаимодействие по модели "запрос–ответ", что позволило протестировать и изолировать бизнес-логику сервиса.
С помощью RabbitMQ реализовано асинхронное взаимодействие между компонентами системы. Producer отправляет сообщения в очередь, а Consumer обрабатывает их независимо, вызывая gRPC сервис. Это обеспечивает слабую связанность компонентов, повышает устойчивость системы и позволяет масштабировать обработку сообщений.
Использование префиксов сообщений позволило реализовать маршрутизацию задач внутри одной очереди и одного обработчика.
Все три задачи (подписка на рассылку, генерация пароля, подсчет гласных) успешно реализованы и корректно обрабатываются системой.
